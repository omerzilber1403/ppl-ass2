\section*{Question 1.1}

No, there is no program in L1 that cannot be transformed to an equivalent program in L11. In other words, \textbf{every program in L1 can be transformed into an equivalent program in L11}.

\textbf{Explanation:}\\
The L1 language does not contain user-defined functions (\texttt{lambda}), local variables, or side-effects other than the \texttt{define} operation itself. In L1, the \texttt{define} structure simply binds a name to a primitive value or the result of an expression. Since there are no side-effects or state changes during runtime, we can take any program that uses \texttt{define} and apply \textbf{substitution}: wherever the defined variable name appears, we replace it directly with the evaluated expression or value it points to.

\section*{Question 1.2}

No, every L2 program can be transformed into an equivalent L21 program. 

Unlike L1, L2 includes lambda expressions, which allow variable bindings through function abstraction and application. For simple non-recursive bindings, any use of \texttt{define} can be directly replaced by wrapping the body in a lambda expression and applying it immediately. 

The main challenge in L2 is \textbf{recursive procedures}, where a function needs to reference its own name. However, even without \texttt{define}, recursion can be successfully achieved in L21 by using the \textbf{Y-combinator} pattern (or the self-passing function technique), where the anonymous lambda expression passes itself as an additional argument to achieve self-reference. 

\section*{Question 1.3}

No, every program in L2 can be transformed into an equivalent program in L22.

\textbf{Explanation:}\\
The syntactic restrictions of L22 reflect the foundational rules of pure \textbf{Lambda Calculus}, which is Turing-Complete. Any L2 program can be systematically rewritten to satisfy L22 requirements using \textbf{Currying}: Functions with multiple arguments are transformed into a nested chain of single-parameter functions.\\
\textit{Example:} \texttt{(lambda (x y) (* x y))} becomes \texttt{(lambda (x) (lambda (y) (* x y)))}.

\section*{Question 1.4}

Yes, there is a program in L2 that cannot be transformed to an equivalent program in L23. 

\textbf{Contradictory Example:}
\begin{lstlisting}[caption={Example of program not transformable to L23}]
(define apply-conditionally 
  (lambda (cond? f g x) 
    (if cond? (f x) (g x))))
\end{lstlisting}

\textbf{Explanation:}\\
In L2, functions are first-class citizens and can be passed as arguments. The crucial point is that the exact function passed to \texttt{f} or \texttt{g} might only be determined dynamically at runtime based on other conditions or user inputs. Since L23 strictly forbids passing functions as arguments, and we cannot predict this dynamic runtime behavior in advance, we cannot statically "hardcode" or inline the specific function calls ahead of time.

Furthermore, we cannot simulate this dynamic behavior using workarounds. In L2, functions act as \textbf{closures} that capture variables from their lexical environment. Because L2 lacks built-in complex data structures (like arrays or lists) to save this captured state, it relies entirely on higher-order functions to represent and store dynamic data. Without the ability to pass functions, L23 has no way to pack, store, or evaluate these closures at runtime, making the transformation impossible without losing the program's core logic.

\section*{Question 1.5}

Special forms are required because standard primitive operators rigidly evaluate \textbf{all} of their arguments before the operator itself is applied (applicative-order evaluation). We cannot define control-flow structures or environment bindings as primitive operators because they require custom evaluation rules—they need the power to control \textit{if}, \textit{when}, or \textit{how} their arguments are evaluated.

\textbf{Example:}\\
Consider the \texttt{if} expression: \texttt{(if (= x 0) 0 (/ 5 x))}.\\
If \texttt{if} were defined simply as a primitive operator, the language evaluator would evaluate all of its arguments first before applying the \texttt{if} logic. If \texttt{x} happens to be \texttt{0}, the program would attempt to evaluate the third argument \texttt{(/ 5 0)} and immediately crash with a \textbf{Division by Zero} error, defeating the entire purpose of the condition. 

\section*{Question 1.6}

\textbf{No, a function body with multiple expressions is not required in pure functional programming.}

\textbf{Explanation:}
\begin{itemize}
    \item \textbf{Pure Functional Programming:} Pure functions have no side-effects (like altering memory or printing). If a function body has multiple expressions, the evaluator calculates the earlier expressions and immediately discards their results, returning only the final expression. Since earlier expressions don't change any state, evaluating them is useless; a single expression is entirely sufficient.
    \item \textbf{Where is it useful?} Multi-expression bodies are essential in \textbf{impure or imperative languages} that support side-effects. In such languages, early expressions perform sequential, meaningful actions (like updating a variable or printing data) before the final value is returned.
    \item \textbf{What about L3?} L3 introduces local bindings (\texttt{let}) and data structures, but it remains a pure functional language with no side-effects. Instead of writing a sequence of separate expressions to calculate intermediate steps, L3 handles this purely by binding intermediate results to local variables using a single \texttt{let} expression. Therefore, a multi-expression body is still not required.
\end{itemize}

\section*{Question 1.7}

A \textbf{Lexical Address} is a static coordinate system used to replace variable names in an Abstract Syntax Tree (AST) with exact pointers to where they are defined. This optimizes the interpreter by removing the need for dynamic string-matching environments during runtime. 

A bound variable's lexical address consists of two coordinates:
\begin{enumerate}
    \item \textbf{Depth:} The number of nested scopes (e.g., \texttt{lambda} contours) the interpreter must traverse outward to reach the variable's declaration. \texttt{0} indicates the immediate current scope.
    \item \textbf{Position:} The zero-based index of the variable within that specific scope's parameter list.
\end{enumerate}

\textbf{Example:}\\
Consider the following nested expression:
\begin{lstlisting}[caption={Nested Lambda Expression}]
(lambda (y x) ((lambda (x) (+ x y)) (+ x z)))
\end{lstlisting}

When transformed to use lexical addresses in the format \texttt{[variable : depth position]}, it becomes:
\begin{lstlisting}[caption={Expression with Lexical Addresses}]
(lambda (y x) ((lambda (x) (+ [x : 0 0] [y : 1 0])) (+ [x : 0 1] [z : free])))
\end{lstlisting}

\section*{Question 1.8}

\textbf{Advantage of the PrimOp representation:}\\
Execution efficiency and direct implementation. When primitive operators are parsed as dedicated syntactic nodes (e.g., \texttt{PrimOp("+")}), the interpreter recognizes them immediately and triggers the underlying native code. This approach completely bypasses the runtime overhead associated with searching through dynamic environments, creating closures, or handling substitution processes.

\textbf{Advantage of the Closure representation:}\\
Architectural uniformity and first-class flexibility. By predefining primitive operators as standard closures in the initial global environment (parsed simply as variable references), the interpreter treats built-in and user-defined functions exactly the same. This simplifies the core evaluation logic by eliminating special-case handling for applications, while allowing primitives to act as first-class citizens that can be seamlessly passed as arguments or bound to new variables.

\section*{Question 1.9}

\subsection*{a. Switching from Applicative Order to Normal Order}
\textbf{Reasons for switching:}
\begin{enumerate}
    \item \textbf{Ensuring Termination (Halting):} In applicative order, all arguments are evaluated before the function is applied. If an unused argument contains an infinite loop or an error, the program will crash or hang. Normal order delays evaluation until the argument is actually needed, allowing the program to terminate successfully if that argument is discarded.
    \item \textbf{Avoiding Unnecessary Computations:} If an argument passed to a function is never used within its body, applicative order wastes processing time evaluating it. Normal order completely skips its evaluation.
\end{enumerate}

\textbf{Example:}
\begin{lstlisting}[caption={Applicative vs Normal Order Example}]
(define loop (lambda () (loop)))
(define f (lambda (func) 5))

(f (loop))
\end{lstlisting}

\subsection*{b. Switching from Normal Order to Applicative Order Evaluation}
\textbf{Primary Reason:}\\
\textbf{Efficiency and Avoiding Redundant Computations.} In pure normal order (call-by-name), if an argument is used multiple times inside the function body, the un-evaluated expression is duplicated. Consequently, the interpreter is forced to evaluate the exact same expression multiple times during execution. 
Applicative order resolves this issue completely by evaluating the argument \textbf{exactly once} before entering the function body and passing a pre-calculated final value, thereby optimizing runtime and saving system resources.

\textbf{Example in L3:}
\begin{lstlisting}[caption={Normal vs Applicative Order Example}]
(define square (lambda (x) (* x x)))
(define expensive-calc (lambda () (+ 5 5)))

(square (expensive-calc))
\end{lstlisting}

\section*{Question 1.10}

\subsection*{a. The Role of \texttt{valueToLitExp} in L3}
\textbf{Role and Purpose:}\\
The function \texttt{valueToLitExp} is a helper function used in the L3 interpreter during the evaluation of expressions under the \textbf{Substitution Model}. Its primary role is to convert a computed runtime \textbf{\texttt{Value}} (such as \texttt{NumVal} or \texttt{BoolVal}) back into a literal expression (\textbf{\texttt{LitExp}}), which is a node type in the Abstract Syntax Tree (AST).

\textbf{Why it is necessary:}
\begin{itemize}
    \item \textbf{Type Mismatch in the AST:} In the substitution model, when a function application (\texttt{AppExp}) is evaluated, the arguments are first reduced to runtime \texttt{Value} objects. The interpreter must then substitute these evaluated results into the function's body.
    \item \textbf{AST Expects Expressions (\texttt{Exp}):} The function body is represented as an AST composed entirely of expressions (\texttt{Exp}). You cannot directly inject a raw runtime \texttt{Value} (like \texttt{NumVal(5)}) into an AST structure that strictly expects an \texttt{Exp} subtype.
    \item \textbf{The Solution:} \texttt{valueToLitExp} bridges this operational gap. It takes the runtime \texttt{Value} and maps it to its corresponding AST literal expression representation (e.g., mapping \texttt{NumVal(5)} to \texttt{LitExp(5)}). Since \texttt{LitExp} is a valid \texttt{Exp}, the substitution algorithm can safely replace variable references (\texttt{VarRef}) with this literal expression node throughout the function body's AST.
\end{itemize}

\subsection*{b. Why \texttt{valueToLitExp} is not needed in the Normal Evaluation Strategy (\texttt{L3-normal.ts})}
In the normal order evaluation strategy (call-by-name), arguments are passed into a function \textbf{without being evaluated beforehand}. 
\begin{itemize}
    \item Instead of reducing an argument to a runtime \texttt{Value} object (like \texttt{NumVal}), the interpreter takes the raw, un-evaluated argument expression directly from the AST (which is already of type \texttt{Exp}).
    \item The substitution mechanism then replaces the parameter variables within the function body's AST directly with these \texttt{Exp} nodes.
    \item Since the process involves substituting an \texttt{Exp} with another \texttt{Exp}, the interpreter never manipulates runtime \texttt{Value} objects during the substitution phase. Therefore, converting a \texttt{Value} to a \texttt{LitExp} is completely obsolete.
\end{itemize}

\subsection*{c. Why \texttt{valueToLitExp} is not needed in the Environment-Model Interpreter}
The environment-model interpreter \textbf{completely eliminates the substitution mechanism} from the evaluation process, keeping the AST structurally immutable during runtime.
\begin{itemize}
    \item In the substitution model, we are forced to inject evaluated results back into the function body's AST, causing a type mismatch that requires \texttt{valueToLitExp}. 
    \item In the environment model, when a function application is evaluated, the interpreter creates a new environment frame (a binding map). This frame maps the parameter names (strings) directly to their computed runtime \texttt{Value} objects.
    \item When evaluating the function body, whenever the interpreter encounters a variable reference (\texttt{VarRef}), it dynamically looks up the variable's name in the current environment and retrieves its \texttt{Value}.
    \item Because runtime values are stored externally in environments rather than being spliced back into the AST node tree, there is no need to transform a \texttt{Value} into a \texttt{LitExp}.
\end{itemize}

\section*{Question 1.11}

\subsection*{a. Why Renaming is Not Required in the Environment Model}
Renaming is completely unnecessary in the environment model because it \textbf{eliminates the substitution mechanism}, meaning the AST remains structurally immutable during runtime, preventing any possibility of variable capture.
\begin{itemize}
    \item In the substitution model, evaluating code involves splicing expressions into function bodies, which risks "capturing" free variables if a local parameter happens to share the same name. To prevent this, strict renaming rules are required.
    \item In the environment model, when a function (closure) is defined, it captures a reference to its definition-time environment. When that closure is applied, a new environment frame is created, which links back to the closure's captured environment.
    \item Variable resolution is performed dynamically at runtime by traversing this static chain of environment frames. Since variables are resolved by looking up names within distinct, isolated memory scopes rather than modifying the text of the program, there is no physical overlap or textual merging of expressions. Consequently, free variables are always safely resolved in their proper lexical scope, entirely avoiding variable capture and making renaming obsolete.
\end{itemize}




\subsection*{b. Is Renaming Required in the Substitution Model When the Substituted Term is Closed?}
\textbf{No, renaming is not required} in the substitution model if the term being substituted is a closed term.
\begin{itemize}
    \item \textbf{What is a closed term?} By definition, a closed term is an expression where all variables are locally bound, meaning it contains \textbf{no free variables}.
    \item \textbf{When is renaming needed?} In the substitution model, renaming is only triggered to prevent "variable capture." This capture occurs exclusively when a \textit{free} variable inside the injected expression happens to share the same name as a local parameter in the target function, causing it to be accidentally bound to the wrong scope.
    \item \textbf{Why is it safe here?} Since the expression we are substituting is completely closed, it has absolutely no free variables that could clash with or be captured by other parameters. Without any free variables, the risk of capture is non-existent. Therefore, the substitution is perfectly safe and precise without ever needing to activate the renaming mechanism.
\end{itemize}

\section*{Question 2.d: Transformed Program}

Below is the equivalent program where the \texttt{ClassExp} representing the \texttt{circle} has been syntactically transformed into nested \texttt{ProcExp}s (lambdas) with a conditional dispatch chain, as specified in section 2.c.

\begin{lstlisting}[caption={Transformed Scheme Program}]
(define pi 3.14)

(define square (lambda (x) (* x x)))

(define circle
  (lambda (x y radius)
    (lambda (msg)
      (if (eq? msg 'area)
          (* (square radius) pi)
          (if (eq? msg 'perimeter)
              (* 2 pi radius)
              'error)))))

(define c (circle 0 0 3))

(c 'area)
\end{lstlisting}

\section*{Explanation of Changes}
\begin{itemize}
    \item \textbf{Outer Lambda:} The \texttt{class (x y radius)} expression is transformed into an outer \texttt{lambda (x y radius)} which accepts the fields as arguments.
    \item \textbf{Inner Lambda:} It returns an inner closure \texttt{lambda (msg)} that acts as the object's method dispatcher.
    \item \textbf{Dispatch Chain:} Since the methods in this section have no parameters, the inner body evaluates directly to the first expression of each method's body based on the condition:
    \begin{itemize}
        \item If \texttt{msg} is \texttt{'area}, it executes \texttt{(* (square radius) pi)}.
        \item If \texttt{msg} is \texttt{'perimeter}, it executes \texttt{(* 2 pi radius)}.
        \item Any unknown message falls back to the symbol \texttt{'error}.
    \end{itemize}
\end{itemize}
